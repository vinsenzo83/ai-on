17Key Risks

Risk 1 — UI Significantly Ahead of Backend
The frontend presents a fully capable AI orchestration platform. The backend falls back to mock responses for all AI calls. A user interacting with the live product may not realize they are receiving mock output.
Risk 2 — Critical Config Lost on Restart
API keys, model whitelist, and task priority are all in-memory. Any server restart, crash, or scaling event silently resets all operator configuration with no warning to the user.
Risk 3 — Cost Data Is Not From Real Usage
62 cost records appear to be seeded/test data (5 timestamp clusters, no corresponding job records). Cost dashboards may give a false impression of operational billing history.
Risk 4 — Orchestration Architecture Not Verified as Active
MasterOrchestrator, ComboOptimizer, DynamicPlanner (3,000+ lines) are code artifacts. No evidence was found that they are called during a normal user task execution flow.
Risk 5 — Homepage Exposes Internal Architecture
The homepage explicitly names "ComboOptimizer," "모델 조합," and "자동 선택" in primary navigation. This exposes implementation architecture before the capability is actually operational.
Risk 6 — Single-Provider Dependency, Currently Zero Functional
The operator uses only OpenAI via proxy. The current key is returning 401. Zero functional AI providers are active in the current deployment.
Risk 7 — No Production Safety Net
Default JWT secret is hardcoded. No env-var enforcement prevents insecure deployment. Mock fallback is silently active with no user-visible indicator.

20Summaries

Founder Summary

The product has a compelling concept, a professional UI, and a working server scaffold. The core value proposition — intelligent multi-model orchestration — exists as working code structure but is not operationally verified. The immediate ceiling is a broken API key, not a missing feature. With a working key and 2–3 weeks of persistence work, this could become a genuinely demonstrable system.

Product Team Summary

The feature surface area is large and well-designed. However, the gap between what the UI presents and what the backend delivers is significant. Three high-priority quick fixes would dramatically change the product's credibility: (1) a valid API key, (2) DB persistence for admin config, (3) confirmed E2E execution of at least one pipeline. Until these are done, demos should explicitly acknowledge mock mode.

Engineering Summary

The codebase is large (3,216-line server, 9-file orchestration layer, 11 pipelines) but has structural debt. Key issues: in-memory-only config that rebuilds on every restart; orchestration layer whose connection to HTTP execution path is unclear; benchmark system with 2 entries of uncertain provenance; visionPipeline.js with its real API call explicitly commented out. The multimodalPipeline.js → aiConnector.callVision() path is the cleanest and should be used as the reference for fixing other pipelines.
