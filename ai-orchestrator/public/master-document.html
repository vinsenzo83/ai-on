01 문서 목적

이 문서는 AI 오케스트레이터 프로젝트에서 실제로 구현된 것과 그렇지 않은 것을 구분합니다.

14 기술 현실 평가

코드 규모 vs 실제 작동 비율

전체 소스코드 약 15,000줄 이상이 존재하지만, 유효한 AI API 키 없이는 핵심 기능(LLM 추론, 파이프라인 실행)이 전혀 작동하지 않습니다.


[실제 동작하는 것] ✅ 인증/인가 (JWT + RBAC) ✅ 감사 로그 DB 쓰기 ✅ 사용자 CRUD ✅ API 키 등록 UI + 테스트 엔드포인트 ✅ 파이프라인 메타데이터 반환 [코드 존재하지만 미작동] 🟡 multimodalPipeline.js → callVision() 호출 시도 but 401 🟡 모델 화이트리스트/우선순위 설정 → 인메모리만 ❌ 모든 LLM 추론 → _mockResponse() 대체 ❌ 비동기 작업 큐 (BullMQ/Redis 미연결) ❌ 실시간 WebSocket 메트릭 (502)

16 현재 상태 리스크
🔴 리스크 #1 — 핵심 기능 미작동

UI는 완성되어 있으나 AI 추론이 전혀 작동하지 않음. 사용자가 파이프라인을 실행하면 mock 응답이 반환되어 허위 결과를 제공할 위험.

🔴 리스크 #2 — 설정 비영속성

서버 재시작 시 API 키, 모델 화이트리스트, 우선순위가 초기화됨. 운영 환경에서 치명적.

🟡 리스크 #3 — JWT 시크릿 하드코딩

src/server.js에 JWT 시크릿이 평문으로 하드코딩. 보안 취약점.

🟡 리스크 #4 — 비용 데이터 신뢰성

62건의 비용 레코드가 실제 사용 데이터인지 seed 데이터인지 불명확. 잘못된 비용 보고 위험.

🔵 리스크 #5 — 코드와 실제 간 괴리

3,000줄 이상의 오케스트레이터 코드(masterOrchestrator, comboOptimizer 등)가 실제 실행 경로에 포함되는지 미확인. 데드코드 가능성.

17 실제 SaaS라 부르기 위한 최소 조건
  • 1️⃣
    유효한 AI API 키 1개 이상 등록 → mock 모드 해제, 실제 LLM 추론 결과 확인.
  • 2️⃣
    API 키·모델 설정 DB 영구 저장 → 재시작 후 설정 복원. DB에 api_configs 테이블 추가 필요.
  • 3️⃣
    엔드-투-엔드 파이프라인 1개 검증 → 요청 → 실제 LLM → 비용 기록 → DB jobs 저장 → UI 결과 표시.
  • 4️⃣
    JWT 시크릿 환경변수 강제 → process.env.JWT_SECRET 없을 시 서버 시작 거부.
  • 5️⃣
    WebSocket 실시간 메트릭 정상화 → 502 오류 해결, 실제 지표 전송.
18 최종 평가

💼 창업자 관점

제품 외형과 UI는 완성도가 높습니다. 그러나 핵심 가치제안인 "AI 오케스트레이션"이 아직 mock 모드입니다. 유효한 API 키 1개와 DB 영속성 코드 추가만으로 데모 가능 수준으로 전환할 수 있습니다.

🧑‍💼 제품 관점

어드민 패널은 사용자 관리, 비용 조회, 감사 로그 등 운영 도구로서 기본 기능을 갖추고 있습니다. 그러나 API 키와 모델 설정의 비영속성 때문에 실제 팀이 운영하기에는 불완전합니다.

🔧 엔지니어링 관점

코드 구조는 잘 설계되어 있으며 멀티-프로바이더 라우팅 아키텍처는 확장 가능합니다. 실제 검증이 필요한 항목: (1) masterOrchestrator 호출 경로, (2) BullMQ/Redis 연동, (3) 모든 stub 파이프라인의 실제 구현, (4) JWT 시크릿 하드코딩 해결, (5) 모델 레지스트리 DB 연동.
